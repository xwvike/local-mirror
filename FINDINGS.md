## 1. 符号链接导致路径穿越 / 信息泄露【安全，最高优先级】

### 现象
同步目录内放一个指向同步根目录**之外**任意可读文件的符号链接，reality 服务端会
解引用该链接并把目标文件内容发送给任何连接上来的客户端。

### 复现步骤与证据
```bash
# Mac 端（reality，同步目录 /tmp/lm-test/mac-reality）
ln -s /etc/hosts symlink-to-outside.txt
```
6 秒后，debian 端（mirror）收到的 `symlink-to-outside.txt` 内容：
```
##
# Host Database
...
127.0.0.1	localhost
```
与 Mac 本机 `/etc/hosts` **逐字节一致**。该文件在 debian 端落地为一个**普通文件**
（不是符号链接），mtime 是 `/etc/hosts` 本身的最后修改时间（5月1日），证实是被
完整解引用读取后原样传输的。

另一个案例：`ln -s medium.bin symlink-to-medium.bin`（指向同步目录内的文件）也被
解引用为一份 3MB 的独立文件副本，符号链接语义完全丢失（这个案例本身不算信息
泄露，但同样说明符号链接从未被当作"链接"处理，而是被当作目标内容处理）。

### 根因
- `internal/tree/buildFileTree.go`：`filepath.WalkDir` 对符号链接是 `Lstat` 语义
  （不解引用），但 worker pool 里对文件调用 `utils.CalcBlake3`（`pkg/utils/helper.go:52`
  内部用 `os.Open`，**会解引用**）计算哈希。
- `internal/network/server.go:351`：`handleFileRequest` 用 `os.Stat`（**解引用**）
  取文件大小和存在性；此前实现的路径穿越校验（`server.go:347-349`）只检查了
  **客户端请求路径字符串**本身是否包含 `..`，完全没有检查该路径在磁盘上
  `Lstat` 之后是否是符号链接、以及链接目标解析后是否仍在同步根目录内。
- 结果：只要同步目录里存在一个指向目录外任意可读文件的符号链接（无论是用户
  手滑创建的，还是恶意放置的），服务端会毫无察觉地把该文件内容发给任何握手
  成功的客户端。

### 影响范围
只要能连上 reality 服务端（默认无认证；`-k` 加密只保证传输保密和双向认证，
不限制"能读什么"），且同步目录内存在一个跨界符号链接，就能读到运行 reality
进程的用户有权限读取的**任意文件**。

### 修复
**方案：符号链接完全不追踪**（不解引用、不同步、不删除），而不是"允许同步但
限制在根目录内"——后者仍需要对每次访问做实时的路径解析和边界检查，多一层
可能出错的逻辑；直接跳过则连"检查"这一步都不需要，攻击面为零。

- `internal/tree/buildFileTree.go`：`WalkDir` 回调里用 `d.Type()&fs.ModeSymlink`
  （Lstat 语义，不解引用）识别符号链接，命中则跳过（不建树、不算哈希）。
- `internal/watcher/eventFilter.go`：原来对 Create/Write 事件用 `os.Stat`（解引用）
  取文件信息，现在先用 `os.Lstat` 判断是否为符号链接，是则跳过；确认不是链接
  后再走原有的 `os.Stat` 逻辑处理真实文件/目录。
- `internal/network/server.go`：`handleFileRequest` 里增加纵深防御——即使目录树里
  不该出现符号链接节点，仍对请求路径做一次 `os.Lstat` 检查，命中符号链接直接
  拒绝该请求。三层防护中任意一层失守，其余两层仍能兜住。

实测：`ln -s /etc/hosts leak.txt` 后，服务端建树日志打印"跳过符号链接"，客户端
最终目录里不出现该文件、更不会泄露 `/etc/hosts` 内容。

## 2. 符号链接删除无法被实时检测【正确性】

### 现象
一个已经同步过的、有效的符号链接被删除后，这个删除**可能长时间（实测超过
2 分钟未同步，理论上限最长 30 分钟）**不会被镜像到客户端，直到全量扫描安全网
兜底纠正。普通文件（无论单个还是批量）的创建/删除都能被正常实时检测，问题
精确定位在符号链接场景。

### 复现步骤与证据
```bash
ln -s medium.bin re-symlink-1.bin
ln -s /etc/hosts re-symlink-2.txt
# 等待同步完成后
rm re-symlink-1.bin re-symlink-2.txt
```
2 分钟内连续轮询 debian 端，残留文件数始终是 2（未减少）。对照组：3 个**普通
文件**同时创建再同时删除，同步完全正常、无延迟。

### 根因
`internal/watcher/eventFilter.go:69-81`：当 fsnotify 报告一个 `Create`/`Write`
类型事件、且 `os.Stat`（解引用）判定目标是目录时，代码会：
1. 把这个"目录"节点重新 upsert 落库
2. 调用 `scanNewDirContents(event.Name)`（`eventFilter.go:113`），**单向**
   `os.ReadDir` 该目录、为每个**当前存在**的条目合成一个 `Create` 事件

这个分支设计初衷是应对 "watch 建立前就已写入的新目录内容"（如 `mkdir -p`）。
但推测在 macOS 的 kqueue 后端下，符号链接的删除操作触发的 vnode 事件，被
fsnotify 报告为父目录级别的笼统事件（而不是针对该符号链接的 `Remove` 事件），
从而落入了这个"目录"分支。`scanNewDirContents` 只会为磁盘上现存的条目生成
`Create` 事件，**从未反向比对"数据库记录中存在、但磁盘上已经消失"的条目**，
所以这类删除完全不会被这条路径检测到。

唯一的兜底是：
- tier2 冷目录轮询（`hasDirectoryChanged`，`scoreWatch.go`）—— 如果该目录恰好
  被评为冷门目录，可能在几十秒到几分钟内被轮询发现
- 客户端全量扫描安全网（默认 1800 秒 = 30 分钟）

对于长期处于 tier1（活跃、实时监听）的目录，符号链接删除可能要等到 30 分钟一
次的全量扫描才能被纠正，与"实时镜像"的设计目标有明显落差。

### 修复
作为 #1 修复的直接结果一并解决：符号链接现在从建树阶段起就完全不被追踪，
`scanNewDirContents` 也不会为符号链接合成 `Create` 事件（`os.Lstat` 先过滤掉了）。
既然符号链接从未进入数据库树，也就没有"数据库有、磁盘没有"这类需要被删除
检测捕捉的状态——问题从根上消失，没有再单独修 `scanNewDirContents` 的单向扫描
逻辑本身。

**注意**：`scanNewDirContents` 单向扫描、不反向比对删除的这个设计缺口，理论上
不止符号链接一种触发路径——如果未来在其他场景下遇到"目录级别笼统事件被误判
为创建"的情况，同样的删除检测缺失可能重现。这次修复解决的是已知的具体触发
路径（符号链接），没有修复这个设计缺口本身；是否要重构
`scanNewDirContents` 使其具备双向比对能力，留待下次遇到新触发场景时评估。

## 3. 下载写入失败会污染整个连接【稳定性/正确性】

### 现象
客户端下载文件时，如果本地写入失败（磁盘满、权限问题、`ulimit -f` 等任何
`os.File.Write` 返回错误的场景），会导致这个 TCP 连接的**协议字节流永久错位**，
后续在该连接上发送的所有请求都可能读到残留的、不属于当前请求的字节，被
错误解析成随机的消息类型，引发短时间内大量无退避的失败重试。

### 复现步骤与证据
在 debian 端用 `ulimit -f 1024`（限制最大文件大小约 512KB）启动一个客户端，让它
去同步一个 3MB 文件。日志显示：

```
[error] 任务执行失败 变更追踪: invalid recent change response message type, got 5
[error] 任务执行失败 变更追踪: invalid recent change response message type, got 3
[error] 任务执行失败 变更追踪: invalid recent change response message type, got 4
[error] 任务执行失败 变更追踪: invalid recent change response message type, got 9
... （共 39 次，全部发生在 20 毫秒内，无任何退避间隔）
```
`.local-mirror/partial/` 目录下正确保留了一个 1,048,576 字节（正好是 ulimit 限制
的边界）的 `.part` 分片文件——证实是文件写入在中途被 `ulimit` 截断失败。

最终这次是"侥幸自愈"的（约 50 秒后恢复正常的长轮询节奏），但过程中的雪崩式
重试没有任何设计上的保证一定能恢复。

### 根因
`internal/network/client.go:510`：
```go
if _, err := file.Write(dataMsg.Data); err != nil {
    return "", fmt.Errorf("error writing file data: %w", err)  // 未包装 appError.ErrConnection
}
```
这个错误没有被标记为连接级错误。但它发生的时机是**协议文件数据流的中间**——
服务端此时仍在按照 `FileResponse.FileSize` 持续发送 `FileData`/`FileComplete`
消息，客户端却提前 `return` 放弃了这次下载，且**没有排空（drain）这个连接上
仍在传输中的剩余数据、也没有关闭连接**。连接对象被原样返回给调用方继续复用。

对比：同一份代码里，"分片过期需要重新下载"的场景（`client.go` 附近的
`drainFileSession`）已经正确实现了排空逻辑，但这条路径没有复用它。

下一次在这个"脏"连接上发起任何请求（例如变更追踪的长轮询），`receiveMessage`
读到的第一批字节实际上是上一次未消费完的 `FileData`/`FileComplete` 消息残留，
被当作新请求的响应头解析，`Type` 字段落在了协议里其他消息类型的数值上，于是
产生"invalid recent change response message type, got N"这类错误。

### 影响
任何导致下载中途失败的原因（真实磁盘满、目标路径权限变化、进程收到需要
提前退出的信号等）都可能触发。已知的雪崩式重试本身也是一种资源风险
（虽然本例中因为都是本地解析失败，不产生额外网络流量，但会占用 CPU）。

### 修复（第一轮，2026-07-08，本地 loopback 验证）
`internal/network/client.go`：`DownloadFile` 里两处会在数据流中间发生的错误——
`file.Write` 失败、以及 `FileData`/`FileComplete` 消息的 `SessionID` 不匹配——
统一包装成 `appError.ErrConnection`。这让上层（`processFileDiff` →
`getDirectory` → `drainNextLevel`）能识别出"这个连接已经不可信"，从而触发
`fileClient.Reconnect()` 而不是带着一个字节流错位的连接继续发送后续请求。

**修复过程中发现的新问题——重连风暴**：把写入失败标记为连接错误后，用
`ulimit -f` 复现测试时观察到 **1 秒内触发了 1359 次重连**。`drainNextLevel`
原来的逻辑是"重连成功就立刻无延迟地重试同一个目录"，若失败原因是持续性的
（如 `ulimit`），会形成没有退避的忙循环。一并加了目录级失败计数
（`maxDirRetries = 3`）与递增退避，本地验证重连次数从 1359 降到 4。

**这一轮的验证结论后来被证明不完整**——见下方 2026-07-09 的更新。

### 修复（第二轮，2026-07-09，远程真实网络复测发现并修复）

用真实网络（而非 loopback）+ 真实的多文件目录重跑同一个 `ulimit -f` 场景，
发现第一轮的结论是错的：**目录级别放弃的粒度太粗，会拖累同目录内其他完全
正常、不受 ulimit 影响的文件**——`still-syncing.txt`、`sub/file.txt` 这类
几字节的小文件，只因为和 `medium.bin`（持续失败）在同一个目录的 diff 列表里，
每轮重试都卡在 `medium.bin` 上，连续 3 次后整个目录被放弃，这些正常文件
自然也没有机会被处理，需要等下一轮全量扫描才可能补上。

**修复**：把"目录级别重试"和"单个文件持续失败"解耦。`getDirectory` 新增
`itemFailures`/`blacklist` 两个参数（由 `drainNextLevel`/`TrackingChanges`
创建并跨多次目录重试维持状态）：处理 diff 时先检查具体路径是否已被拉黑，
拉黑则跳过（不再尝试，但不影响同目录其他项）；某项连续失败超过
`maxItemRetries = 3` 才拉黑。目录级别的失败计数只在"没有新增黑名单项"（即
完全没有进展）时才递增，一旦本轮拉黑了新文件就重置为 0——避免目录内有
多个问题文件时，目录级重试预算在文件被逐个拉黑之前就被耗尽。

**同一轮里又挖出了更深的根因——重连后的连接在服务端从未被真正注册**。
排查"文件被正确拉黑后，为什么同目录的其他正常文件还是同步不了、且反复报
`failed to receive message: EOF`"时发现：`FileClient.Reconnect()` 内部调用
的 `Reverify()` 发送的是 `MsgTypeReverify`，这个消息体是**空的**——服务端
`handleReverify` 收到后直接応答，全程不知道"是哪个 InstanceID 的客户端"，
因此也就**从未把这次重连的新连接注册进 `clientMap`**（服务端代码里甚至有
一行原有注释直接写明"重连后的新连接尚未握手，clientMap 中没有记录"）。
后续任何依赖 `clientMap.Load(client.ID)` 的请求（`TreeRequest`/`FileRequest`
等，此时本地 `client.ID` 还是零值）都会被判定为"client not found"
（这本身被包装成 `appError.ErrConnection`），服务端主动关闭连接，客户端读到
EOF——**这是一个从项目最初就存在、但因为此前几乎没有代码路径会触发
`Reconnect()` 之后立即发起业务请求，而从未被真正触发过的根本性设计缺陷**；
这次因为 #3 的修复让"写入失败"也会触发 `Reconnect()`，才第一次被引出水面。

**真正的修复**：把 `Reverify()` 改为发送真正的 `MsgTypeHandshake` 消息
（复用服务端已有的、会执行 `clientMap.Store` 的 `handleHandshake` 处理
逻辑），客户端收到握手响应后仍按原语义校验其中的 `ServerID`/`Version`
是否与已知值一致（不一致说明连到了不同的服务器，返回错误触发整个会话
重建）——`MsgTypeReverify`/`handleReverify` 相关代码目前变成死代码，
未清理（不影响功能，留给下次一并处理）。

**实测**（真实网络，重新验证，两处修复叠加后）：`ulimit -f 1024` 场景下，
`still-syncing.txt`、`greeting.txt`、`sub/file.txt` 全部正确同步，只有
`medium.bin` 被精确拉黑（日志："medium.bin 连续失败 3 次，本轮放弃同步该项
（不影响目录内其他文件）"）；重连次数 4 次；EOF 错误、"client not found"、
协议乱码错误均为 0。随后完整回归：allow-delete 真删除、重命名检测、500MB
文件 81MB/91MB 处两次 kill -9 断点续传（MD5 完全一致）、跨机器 Noise 加密
同步与错误口令拒绝，全部通过。

## 4. 镜像对源端异常没有任何删除保护【设计风险，用户最初的担忧已证实】

### 现象
如果 reality 服务端的同步目录因为任何原因（路径参数配置错误、挂载点未挂载、
磁盘故障、误操作等）意外变空或大幅缩减，所有下游镜像客户端会在**下一次全量
扫描或重连对账中，无条件地把本地"服务端没有"的文件全部删除**，过程中没有
任何数量级异常检测、没有确认步骤、没有限流或熔断。

### 复现步骤与证据
1. debian 端已镜像 10 个文件/目录（此前测试积累的沙箱数据）
2. 停止 Mac 端 reality 进程，改用一个全新创建的**空目录**重新启动（模拟路径
   配置错误）
3. debian 客户端因连接断开触发重连（新进程 = 新 InstanceID），日志：

```
14:46:59.979  已连接服务端 10.10.0.5:52345（新 InstanceID: 1988962190）
14:47:00.019  Received tree response ... data length: 2 bytes   ← 服务端返回空数组 []
14:47:00.020  Diff count for .: 10                              ← 10 个文件全部判定为待删除
14:47:00.023  Full scan completed, total time taken: 44.323385ms
```

**从服务端目录变空到本地 10 个文件全部清空，总耗时约 44 毫秒**（不含重连本身
约 5 秒的退避等待）。

### 定性
这不是代码 bug——这是"忠实镜像"设计哲学的必然推论：客户端的职责就是让自己
的状态与服务端汇报的状态一致。但从数据安全角度看，这意味着**任何让源端暂时
或永久"看起来"内容大幅减少的原因，都会在几十毫秒内无条件传导到所有下游**，
没有任何缓冲让人介入或补救。这正是最初提出"镜像功能可能一不小心让文件全部
消失"这个担忧所指向的真实机制，现已通过实验证实。

### 修复（2026-07-08 讨论确定并实现）

与用户讨论后确定的方案，思路上与 rsync 的成熟先例一致——rsync 默认**不**删除
目标端多出的文件，必须显式传 `--delete` 才会启用；这里采用同样的"默认安全，
危险行为需显式解锁"模式：

**1. 默认改为"只增量同步，不删除"（sync-only / additive 模式）**
   客户端默认只做 create/modify，服务端没有但本地有的文件不再自动删除。这直接
   消灭了 #4 描述的最灾难性后果——源端异常清空不会再级联删除下游。

**2. 完全"忠实镜像"（含删除）需要显式解锁参数**
   例如 `--allow-delete` 或类似的显式开关。不加这个参数，即使服务端目录真的
   变空，本地文件也不会被清空，只会停止收到新内容——行为上是"安全降级"而不是
   "静默破坏"。

**3. 解锁删除时，额外的目录白名单/黑名单校验：拒绝在"关键路径"上运行**
   即使用户显式加了删除解锁参数，如果同步根目录（`-p` 解析后的路径）命中一份
   维护的"危险路径"列表，则直接拒绝启动并报错，不给开枪的机会。需要覆盖
   macOS / Linux / Windows 三个平台各自的关键路径，例如：
   - 用户主目录本身（`$HOME` / `%USERPROFILE%`）
   - 文件系统根目录（`/`、卷根 `C:\`）
   - 系统关键目录：`/etc`、`/System`、`/Library`（macOS 系统级）、`/bin`、
     `/usr`、`/var`（Unix 类）；`C:\Windows`、`C:\Program Files`、`C:\Users`
     （Windows）
   - 该列表需要能持续补充维护，而不是一次性写死

**4. 关键约束：必须是路径匹配，不能是目录名匹配**
   校验的对象是**解析后的绝对路径**（且需处理符号链接，参考发现 #1 的教训，
   应该用类似 `filepath.EvalSymlinks` 取真实路径后再比较，防止用符号链接绕过），
   而不是简单检查路径的某一段名字是否等于 "etc"、"home" 之类的字符串。否则
   会把正常的项目子目录（如 `~/project/etc/nginx.conf.d`）误杀，也完全防不住
   `ln -s /etc dangerous-alias && local-mirror -p dangerous-alias` 这种绕过方式。
   判定逻辑上，不仅要检查同步根目录是否**等于**某个危险路径，还要检查是否是
   某个危险路径的**祖先目录**（例如把同步根设为 `/` 或 `~` 本身，会把 `/etc`、
   `/System` 等都纳入其管辖范围，同样必须拒绝）。

这套方案把 #4 从"设计风险"变成"默认安全、危险行为需要用户主动、明确地选择"，
且加了一层与用户操作意图无关的兜底（关键路径拒绝），双重保险。

**实现**：
- `config.AllowDelete`（`--allow-delete`，默认 `false`）。`internal/mirror.go`
  的 `processDiffItem` 在 `action == "delete"` 时，未开启则直接跳过并记 debug
  日志，不做任何本地删除。
- 新增 `internal/safety` 包（`paths.go`）：`dangerousPaths()` 按 `runtime.GOOS`
  返回 darwin/windows/其余类 Unix 各自的关键路径列表（含用户主目录、文件系统
  根、`/etc` `/System` `/usr` 等系统目录，`C:\Windows` `C:\Users` 等）；
  `CheckDeletableRoot` 用 `filepath.EvalSymlinks` 解析真实路径后，检查是否
  等于某个关键路径、或是某个关键路径的祖先目录，命中则拒绝。
- `cmd/local-mirror/main.go`：仅当 `--allow-delete` 且当前模式会同步（mirror/
  relay）时才做这项校验，校验失败直接 `os.Exit(2)`，不会进入运行状态。
- 启动横幅新增"删除同步"一行，明确展示当前是"关闭（仅增量）"还是"开启
  （忠实镜像）"，避免用户不清楚当前处于哪种模式。

实测：`$HOME` 目录下加 `--allow-delete` 被正确拒绝；`ln -s /etc alias &&
--allow-delete -p alias` 这种符号链接绕过尝试，被 `EvalSymlinks` 解析出真实
路径 `/etc`（macOS 上 `/etc` 本身是指向 `/private/etc` 的链接，也被正确
解析）后同样拒绝；在普通沙箱路径下 `--allow-delete` 正常工作，删除按预期
执行；不加该参数时，服务端删除文件、客户端保留本地副本，不做任何删除。

## 5. 批量并发变更下客户端偶发崩溃【稳定性，根因未定位】

### 现象
创建一个新目录并快速写入 150 个文件后，观察到一次 debian 端 mirror 客户端
进程静默退出（`pgrep` 找不到该进程，仅剩父 shell）。日志在 "开始执行任务:
变更追踪" 之后戛然而止，没有任何后续记录、没有 panic 堆栈（由于 ssh 命令的
输出重定向在该次调用中存在问题，不能完全排除是日志丢失而非真正无输出）。

### 复现尝试
用完全相同的操作（新建目录 + 150 文件写入）在修复了日志捕获方式后重新复现，
**未能再次触发**——进程存活、150 个文件通过正常的实时推送路径全部正确同步。

### 现状
判定为间歇性问题，具体触发条件不明确。可能与以下因素有关但未验证：
- 大量文件在极短时间内触发的 fsnotify 事件洪峰与 1 秒创建事件合并窗口
  （`eventFilter.go` 的 `createTimer` 防抖）之间的竞态
- 长轮询响应处理与新一轮请求发起之间的某种时序问题

需要更针对性的压力测试（大量重复次数、更大并发量）才能稳定复现和定位。

## 6. 权限被拒绝的文件无限重试【健壮性，影响较小】

### 现象
服务端上一个客户端无读权限的文件（`chmod 000`），会被正确记录在目录树中，
但每次客户端请求下载都会失败（服务端 `error calculating file hash`）。服务端
处理优雅（不崩溃、不泄露），客户端也优雅报错，但没有观察到任何退避或"放弃
重试"机制——只要这个文件的 diff 仍判定为需要同步，客户端会一直重复请求。

### 影响
不会导致数据损坏或安全问题，但会持续占用请求次数和刷日志，在权限配置不当
的目录里可能造成噪音。

## 7. 全量扫描游标使用客户端时钟【正确性，未能物理复现】

### 现象（代码审查发现，非实测）
`internal/mirror.go:434`：
```go
lastChangeCursor = startTime.Unix()   // startTime 是客户端的 time.Now()
```
全量扫描完成后，下一次增量查询（长轮询）会把这个**客户端时钟**打的时间戳
作为查询起点发给服务端，服务端用它去查询**服务端自己的** `changed_dirs`
时间线（该 bucket 只保留最近 1 小时）。

如果客户端时钟比服务端**快**，这个起点时间戳会落在服务端的"未来"，服务端会
漏掉 `[服务端当前时刻, 客户端时钟起点]` 这段时间窗口内的真实变更，直到下一次
全量扫描（默认 1800 秒后）才能纠正。如果客户端时钟比服务端**慢**，则只是多查
一段无害的重叠窗口。

### 未能复现的原因
测试环境两台机器均由 NTP 同步，时钟偏差在 1 秒以内，不足以触发可观察的窗口
丢失。物理复现需要人为调整某一端系统时钟，因涉及修改远程共享机器的系统级
状态而未执行。

### 修复
`internal/mirror.go`：`fullScan()` 不再用 `lastChangeCursor = startTime.Unix()`
（客户端时钟）设置游标，改为重置 `lastChangeCursor = 0`。下一次
`TrackingChanges` 会以 `[0, 服务端当前时刻]` 发起一次全窗口查询（此时通常是
一次空 diff，因为全量扫描刚同步过），并采用服务端响应里的 `CoveredUntil`
（服务端自己的时钟）重新确立游标。此后游标全程只由服务端时钟推进，不再有
客户端时钟参与，时钟偏差不再可能导致漏查。

无法在这次的测试环境里物理复现原始问题（时钟本就同步），但基本同步/推送/
重命名/断点续传的回归测试确认这个改动没有引入新问题。

## 正面结果（本次测试验证通过、无问题）

以下场景在真实网络（非 loopback）、跨架构（darwin/arm64 ↔ linux/amd64）环境下
验证正常：

- 基线初始同步与实时增量同步（长轮询推送，延迟约 2 秒）

- 空文件、10 层深层嵌套目录
- mtime 保真（下载后本地 mtime 与源文件一致）

- 断点续传：500MB 文件传输至 93MB 时 `kill -9`，重启后正确从断点续传，

  端口扫描正确找到并连接
- 跨机器 Noise 加密（`-k`）：正常同步、错误口令握手被正确拒绝
- 大文件同步与删除的完整性（非符号链接场景）
