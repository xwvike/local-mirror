# 推送化镜像设计（Push Mirror）

> 把"定时轮询的复制器"改造成"实时、省电、休眠友好的镜像"。
> 商定于 2026-07-07。

## 背景问题

原设计客户端每 `-d` 秒（默认 10）轮询一次服务端"最近变了啥"，外加每 30 秒心跳。
带来三个问题：

1. **不实时**：变更最坏要等一个轮询周期才反映到镜像端。
2. **耗电**：客户端每 10 秒一次网络唤醒、服务端 tier2 每 3 秒一次目录轮询，
   永不停歇，笔记本无法进入深度 idle。
3. **休眠不正确**：`changed_dirs` 只保留 1 小时，而增量游标是墙钟时间；
   睡眠超过 1 小时后增量查询会漏掉睡眠期间的变更（仅靠周期全量扫描兜底）。

## 核心决策：长轮询（long-poll）

保留现有**同步请求-响应**协议模型（整个客户端串行化都依赖它），
不引入双连接或全异步多路复用。

客户端发一个"等变更"请求，服务端**不立即回**：
- 区间 `[游标, now]` 内已有变更 → 立刻返回（追赶/重连场景）
- 无变更 → 挂起，直到有新变更（广播唤醒）或挂满上限（保活）后返回

### 为什么不丢通知

长轮询只是"高效闹钟"，**正确性来自时间区间查询**：客户端每次都带
`[游标, now]` 去查持久化的 `changed_dirs`，任何一次唤醒信号丢失都不影响，
下次查询按区间捞回。正常态下游标落后不超过一个挂起周期，稳在 1 小时窗口内。

## 时间参数

| 参数 | 值 | 说明 |
|---|---|---|
| 服务端挂起上限 | 50s | 无变更时返回空包保活 |
| 客户端读超时 | 60s | 必须 > 服务端挂起上限 |
| 服务端空闲断连 | 90s | 覆盖两个挂起周期 |
| 全量扫描安全网 | 30min | 从 5min 放宽（已有实时推送） |
| tier2 轮询 | 30s 起，自适应退避至 5min | 冷目录，从 3s 放宽 |

## 协议变更

线格式不变，但 RecentChange 请求的服务端语义从"立刻回"变为"挂起"。
老服务端会秒回空包导致客户端空转 → 破坏性变更，`ProtocolVersion` 1 → 2，
握手已校验版本，不匹配直接拒。

## 服务端改动

- 变更广播器：`addChangedDir` 落库时发信号，唤醒挂起的请求
- `handleRecentChangeRequest`：有变更立刻回，无变更订阅广播器挂起，
  50s 上限兜底

## 客户端改动

- 变更追踪 ticker → 长轮询循环（发请求 → 60s 读超时阻塞 → 处理/空转）
- 删除 30s 心跳 goroutine（长轮询往返即保活，ServerID 顺带校验）
- 全量扫描降级为低频安全网
- 休眠感知：读返回后比对墙钟跳变，判定刚醒 → 强制全量对账 + 游标重置

## 服务端耗电

tier2 从 3s 放宽到 30s 起，自适应退避（连续无变更指数拉长至上限，
有变更复位）。

## 第二阶段（保真，独立推进）

- `Chtimes`：下载后用服务端 ModTime 补本地 mtime
- 移动检测：同批 diff 中哈希相同的 delete+create 判为 move，本地 rename 不重传
