## 发现 #1【高危】三级安全阶梯的关键路径判定遗漏"子目录"方向

**位置**：`internal/safety/paths.go:103` `IsCriticalRoot` + `internal/safety/paths.go:87` `isAncestorOrEqual`

**根因**：`IsCriticalRoot` 的判定是 `root == d || isAncestorOrEqual(root, d)`，其中 `d` 取自
`dangerousPaths()`（`~`、`/etc`、系统目录等）。`isAncestorOrEqual(root, d)` 检查的是
"**root 是否等于或包含 d**"（例如 root 设为 `/` 会连带覆盖 `/etc`）——这正确处理了
"同步根覆盖关键路径"的方向，但**完全没有检查反方向**：同步根本身是关键路径的
**子目录**时（例如 `~/Documents`、`~/lm-critical-test2`、`/etc/myapp`），`d` 是 `root` 的
祖先而非相反，两个判定条件都为假，`IsCriticalRoot` 判定为"非关键路径"，整个三级
阶梯形同虚设——**零拒绝、零快照备份，直接按普通目录处理**。

**为什么这是最常见的真实场景**：现实中几乎没有人会把同步根直接设成字面意义的
`~` 或 `/etc` 本身；正常用法就是同步到家目录下的某个子文件夹（`~/Pictures`、
`~/projects/mirror-target`……）。这个判定缺口恰好覆盖了"三级安全阶梯"最初设计
要防护的最主要场景，导致该功能在最该生效的地方完全不生效。

**真实复现**（debian，`/home/xwvike/lm-critical-test2` 是家目录子目录）：
```
$ mkdir -p ~/lm-critical-test2 && echo 'ORIGINAL-HOME-FILE-DO-NOT-LOSE' > ~/lm-critical-test2/keep.txt
$ local-mirror -m mirror -p ~/lm-critical-test2 -r <server>   # 不加任何安全旗子
```
结果：**启动零拒绝**（banner 没有"关键路径"行，视为普通目录），几秒后
`keep.txt` 被服务端同名文件**静默覆盖**，且 **`.local-mirror/backups/` 未生成**——
完全没有触发预期的备份保护。对照组：直接 `-p ~`（家目录本身，精确匹配）则正确
拒绝（`exit 2`，报错含 `--allow-critical` 提示）——证明问题精确定位在"祖先方向
检测缺失"，而非整个校验失效。

**影响面**：不仅是这次新增的三级阶梯，`CheckSyncSafety` 是这个判定的唯一使用者，
所以三级安全阶梯（拒绝同步 / `--allow-critical` 解锁 / 覆盖前快照）对"关键路径
子目录"这整类场景**完全不生效**，用户如果把同步根设在家目录下的任意子文件夹，
不会得到文档中承诺的任何保护。

**建议修复方向**：`IsCriticalRoot` 需要双向检测——除现有的
`root == d || isAncestorOrEqual(root, d)`，再加一条 `isAncestorOrEqual(d, root)`
（`d` 是 `root` 的祖先，即 `root` 落在关键路径内部）。两个方向合并即是完整的
"root 与任一关键路径存在包含关系（任一方向）"判定。

**✅ 已修复（2026-07-13，语义经用户确认后细化）**：照字面加双向检测会退化——
`/` 也在危险路径列表里，而一切路径都是 `/` 的子目录，盲加反向检测会让所有路径
都变成关键路径。因此把危险路径分成两类（`internal/safety/paths.go`）：
**系统目录树**（`/etc`、`/usr`、`/var`、`/System` 等）双向检测，`-p /etc/nginx`
这类子目录正确命中；**容器目录**（`/`、`~`、`/home`、`/Users`、`/Volumes` 等）
仅自身命中，子目录（`~/Pictures`）是日常同步目标不受限。这意味着本节复现用的
`~/lm-critical-test2`（家目录子目录）在新语义下**有意不拦**——若每个家目录
子文件夹都要带旗，人人永远带旗，保护即形同虚设；三级阶梯的防护对象收敛为
系统目录树（也正是该功能最初"如果真的需要同步 /etc"的设计动机）。同时修复
`normalize` 对**尚不存在路径**的符号链接前缀解析（逐级解引用最深已存在祖先），
否则 macOS 上 `/etc → /private/etc` 的链接会让 `-p /etc/尚未存在的目录` 漏检。
单测覆盖双向命中/容器子目录放行/不存在子路径/前缀兄弟陷阱；端到端确认
`-p /etc/ssh` 无旗 `exit 2`、`--allow-critical` 放行、家目录子目录零旗可用。

## 发现 #2【中等，性能/响应性】eventFilter 对每个 Write 事件都同步全量重哈希

**位置**：`internal/watcher/eventFilter.go:62`（`CalcBlake3(event.Name)`），
被 `internal/watcher/scoreWatch.go:397` 的 `handleEvents` 单一 goroutine 同步调用
（非 `go eventFilter(event)`，无并发派发）

**根因**：Create/Write 分支对**每一次**触发的 fsnotify 事件都无条件对整个文件重新
计算一次 blake3 哈希，没有防抖、没有"等写入完成再计算"的机制。一个正在被写入的
大文件（流式写入、上传、`cp`/`dd`/日志滚动等）在写入过程中会触发多次 Write 事件，
每次都重新读取该文件**当前已写入的全部内容**来算哈希。由于 `eventFilter` 运行在
唯一的 fsnotify 消费协程里，这个重复哈希计算的耗时会**阻塞整个同步根下所有其他
目录的变更检测**——不仅是这一个大文件的处理被拖慢，是全局的实时性都被拖慢。

**真实复现**（debian → Mac，跨真实网络）：用 `head -c 300000000 /dev/urandom >
huge.bin` 制造一个流式写入的 300MB 文件。观察到客户端一次长轮询请求**卡满了
`LongPollHold` 的 50 秒上限**才拿到变更通知——而该文件在轮询开始后仅 2 秒就已
在磁盘上完成写入（`ls` 显示的 mtime 证实）。这与"服务端迟迟没有广播变更"精确
吻合，指向 `eventFilter` 在这 50+ 秒里一直忙于重复哈希这个不断增长的文件，占满了
唯一的事件消费协程。

**数据正确性未受影响**（重要区分）：`handleFileRequest`（`internal/network/server.go`）
在**每次实际服务文件请求时都重新从磁盘现算哈希/大小**，不依赖这里可能过期的
缓存值；客户端下载完成后自己也会用 blake3 校验完整性。本次复现里，最终下载的
`huge.bin`/`big.bin`/`resume.bin` 全部 MD5 与源文件**完全一致**，重新全量扫描后
也不再出现"反复判定为已修改"的情况（验证过没有因为缓存哈希错误导致的永久重传
循环）。**这是一个响应性/CPU-IO 浪费问题，不是数据损坏或丢失问题。**

**建议修复方向**：给同一文件的重复 Write 事件加防抖（等待一段静默期后只计算
一次最终哈希），或将哈希计算派发到独立 worker（不阻塞事件消费主循环），二者
最好都做。

**✅ 已修复（2026-07-13，防抖 + 移出主循环双管齐下）**：文件的 Create/Write
事件不再当场哈希，改为按路径防抖（`internal/watcher/eventFilter.go`）——事件
只重置该路径的 1 秒定时器，静默期结束后在定时器 goroutine 里做**一次**最终的
stat+哈希+落库，事件消费主循环全程不被哈希阻塞；Remove/Rename 会取消该路径
未触发的防抖。端到端复现原场景：300MB 流式写入进行中，旁边小文件 **3 秒**
同步到达（修复前卡满 50 秒长轮询上限），大文件写完后正常同步、MD5 一致，
重命名/删除传播不受影响。

## 第二轮补充说明：跨国中继链的实现细节

**拓扑**：`hk`（reality，origin）→ `dmit`（relay，中继）→ `bwg`（mirror，末端），
经真实公网、真实跨国路由，全程未触达 Mac 或 debian（`local-mirror` 的同步进程
自始至终只用三台 VPS 各自的公网 IP 互相寻址；我的 SSH 管理连接是 Mac 到各 VPS
的单独控制通道，不参与、也不影响 local-mirror 自身的数据流动）。

**防火墙改动**（经用户明确授权，测试后已完全收回）：三台默认 ufw 拒绝一切入站
（仅放行 22/80/443 及个别既有服务端口），按本次拓扑只需两条最小规则——
`hk` 放行来自 `dmit` 公网 IP 的 `52345:52354/tcp`（`dmit` 的中继要连 `hk`），
`dmit` 放行来自 `bwg` 公网 IP 的 `52345:52354/tcp`（`bwg` 的客户端要连 `dmit`）；
`bwg` 作为链路末端纯客户端不需要任何入站规则。两条规则均限定具体来源 IP，
未向公网开放。测试完成后逐条 `ufw delete` 并核实三台均无残留。

## 结论

两轮、六台真机、多网络环境（本地路由 VPN、阿里云、三个不同国家的独立 VPS
供应商）的系统性测试中，发现两个真实缺陷并**均已于 2026-07-13 修复**（各自
小节内有修复详情）：发现 #1 直接削弱了本项目专门设计的删除/覆盖保护机制的
有效性，修复后系统目录树的子目录正确纳入三级阶梯防护；发现 #2 是不影响数据
正确性的响应性问题，修复后大文件写入不再阻塞全局变更检测。除这两项外，自动发现、YAML
多任务监督、忽略列表、连接上限、加密传输、**真实跨国三跳中继链（含实时传播、
重命名检测、中继节点故障自愈、大文件完整性）**等功能在真实跨机网络下的行为
均与本地/沙箱测试结论一致，没有发现额外的网络环境特定问题。
