# 公网化设计 · v2.0.0 蜕变

> 状态：**三支柱代码均已实现**（分支 feat/v2-public-exposure）——
> C：`--gen-key`/`--show-key`/`--no-encrypt` + 密钥文件解析链（internal/keyfile）；
> B：双栈监听（multiListener）+ JoinHostPort + host[:port]；
> A：四象限（serveConn 剥离 / StartDial / MirrorListen / Role 方向校验）+
> 方向优先 CLI（--send/--receive × --connect/--listen + 位置糖，-m 降级为别名）。
> 未竟：README/YAML 任务的四象限词汇、v2.0.0 发版（见 §D）。
> 目标是**摘掉对 WireGuard 的强依赖，让镜像链路能安全、优雅地跨公网直接跑**。

## 0. 三大支柱与顺序

| # | 支柱 | 一句话 | 角色 |
|---|---|---|---|
| **A** | 传输 / 角色解耦（四象限） | 连接方向与数据方向拆开，谁都能听/拨 | **基座**：改变「谁必须可达」，并让 `mode` 溶解成「方向」 |
| **B** | IPv6 + 域名（name-first，DDNS 外包） | 双栈可达，用域名而非 IP；DDNS 交给用户的客户端 | 摘 VPN 后「跨公网怎么连得上」 |
| **C** | 密钥自管理（强随机 key） | 监听端自动生成强 key，消灭弱口令 | 摘 VPN 后「唯一城墙够不够硬」 |

**依赖顺序**：A 是基座（B、C 的用法都挂在四象限上）；C 独立、非破坏，可最先落地；
B 是承诺特性（不再是可选）。三者齐活即 **2.0.0**。

动机链：摘 VPN → 两层安全塌成一层 → 于是要①让**可达的那端是最合适的机器**（A）、
②**连得上且连得优雅**（B）、③**唯一的城墙是强 key 而非弱口令**（C）。

## A. 传输 / 角色解耦（四象限）—— 基座

### A.1 被焊死的两个方向

当前实现把两个**本应正交**的方向捆成了一个：

- **数据方向**：源 → 汇（单向镜像，产品本质，永不变）。
- **连接方向**：谁发起 TCP（现在锁死为汇拨源）。

结果：**必须公网可达的机器永远是源**。这个耦合限制太大——可达性不该由"你是不是源"决定。

### A.2 四象限

拆开两个方向得到四种组合，当前只开放两格：

| | 监听（listen） | 拨出（dial） |
|---|---|---|
| **源** | ✅ 现在唯一的服务端 | 🆕 **源拨向汇** |
| **汇** | 🆕 **汇监听等源** | ✅ 现在唯一的客户端 |

要补「源+拨 / 汇+听」这一对。其中「**源拨出 → 汇监听**」是核心新用法：本机是源
（本地维护配置），VPS 是汇（挂公网、监听），本机主动拨出去推。

### A.3 为什么这是大赢

1. **本地维护、推向公网 VPS**：本地编辑 → 自动推到 VPS，**永不 ssh 上 VPS 改文件**；
   可扇出成「一份本地源 → N 台监听的 VPS」。
2. **公网故事变简单**：让**监听方 = VPS**（天生有公网 IP），家里那台只**拨出**、出站
   NAT 友好——B 章那堆前置清单对这个用法几乎蒸发。
3. **爆破面更小**：公网暴露的监听端是**备份副本**而非**主数据**，且单向源→汇，
   攻击者无法反写你的源。
4. **relay 降级为组合**（见 A.5），不再是第三种"模式"。

### A.4 可行性：不改报文，中等重构

- **协议报文零改动**：TCP+Noise 全双工，握手后「谁先发第一条请求」与「谁拨号」无关，
  只要**汇在连接建立后仍先说话**即可。
- **Noise `initiator` 自动跟拨号方**（`dialConn` 传 `true`、服务端 `false`），翻转传输后
  自然正确，加密层零改动。
- 要动的是**把「建立连接（拨一个 / accept 循环）」从「在连接上跑源/汇逻辑」里剥出来**，
  让源、汇逻辑都能架在拨出的或被 accept 的连接上；**重连/退避归拨号方**。中等工作量，
  架构上是更干净的分层。

### A.5 CLI：方向优先，让 `mode` 溶解（定稿）

**问题**：`-m reality/mirror/relay` 的心智负担来自两点——把「方向」和「传输」塞进一个词，
还用不透明的名字。**做完 A 之后 `mode` 只剩「方向」这一件事**，于是直说方向即可，
向 rsync「只关注方向、没有模式」的手感靠拢。

**两个正交、直白的轴**（无任何不透明名词）：

```
# 方向轴:--send(我的目录是源) / --receive(我的目录是汇)
# 传输轴:--connect <peer>(我拨) / --listen(我听)

local-mirror -p ./proj   --send    --connect vps.example.net:52345  # 推到 vps(我拨)
local-mirror -p ./backup --receive --connect home.example.net       # 从 home 拉(我拨)
local-mirror -p ./proj   --send    --listen                         # 挂着,等接收方来拨,送出去
local-mirror -p ./backup --receive --listen                         # 挂着,等发送方来拨,收进来
```

**位置糖**（向 rsync 的 `SRC DST` 靠拢，顺序即方向，覆盖最常见的拨号场景）：

```
local-mirror ./proj  @vps.example.net:52345   # 本地→远端 = 推
local-mirror @home.example.net  ./backup       # 远端→本地 = 拉
```

`@addr` 端天然是被拨的一方；`--listen` 两格因"还没有对端可排序"排不进顺序，保留显式旗。
**位置糖管拨号常态，显式两轴管全集（含监听）。**

**握手期方向校验**（rsync 白拿、我们要额外做的一点）：两端是各自独立的 daemon，可能把
**数据方向**配歪（都 `--send` 或都 `--receive`）。好在 `Role` 字段**早已在 v3 握手报文里、
两端也已按 mode 填发**（`client.go` / `server.go` 均 `Role: config.ModeMap[...]`），只是现在
**收到只存不校验**。四象限要做的仅两点：① 让 `Role` 承载**方向（send/receive）**（老值
reality→send、mirror→receive 平滑映射）；② 加**互补校验**——方向不互补即用 v3 **结构化
错误**当场拒（如 `你声明 send，对端也 send`）。

**只校验方向、不校验传输**：传输错配**到不了握手**——都 `--listen` 则无人拨号、连接建不
起来；都 `--connect` 则拨向非监听端直接 connection refused。**传输互补由 TCP 层自然挡下。**
真正会溜进握手又难自查的是"两端都 send"（双方都等对方先要数据 → 静默死等），握手那一查
正好把闷死的 hang 变成秒拒 + 说清原因。

**多半不是 flag-day**：字段现成、老值可映射，大概率无需协议 bump（真要区分新旧语义还有
FeatureBits 兜）。

**relay 溶解为组合**：relay = 一个进程同时做「向上 `--receive --connect` + 向下
`--send --listen`」。保留 `relay` 便捷别名，但背后可解释，不是新概念。

**迁移（非破坏）**：`-m` 保留为**废弃别名**——`reality` ≡ `--send --listen`、
`mirror` ≡ `--receive --connect`；老的 plist / systemd env / YAML `mode:` 不动照跑。
协议层不关心 CLI 用词，**不是 flag-day**。

## B. IPv6 + 域名（name-first / 解析即可 / 不做 NAT）

> 承诺特性。用域名而非 IP 总归优雅，也是应对公网 v6 动态前缀的正解。
> **我们只负责「解析」域名；DDNS 与 NAT 都不插手。**

### B.1 代码现状与缺口

- **reality 监听 `net.Listen("tcp4", "0.0.0.0:%d")`（server.go:64）—— 显式 v4-only，
  今天收不了任何 v6 连接。** 服务端最大缺口。
- **客户端拨号 `fmt.Sprintf("%s:%d", ip, port)`（initConn.go:40）—— 对 v6 字面量会炸**
  （需 `net.JoinHostPort` → `[::1]:52345`）。
- `-r`（`RealityIP`）其实早收域名，正名为 address / host。

代码 delta 三处：**双栈监听（`tcp`/`::`）、`JoinHostPort`、`-r` 当 host**。域名 + 双栈 +
Happy Eyeballs 由 `net.DialTimeout("tcp", …)` 自带。

### B.2 name-first：域名优先

- 对外一律以**域名**为主要寻址形式，IP 字面量仍支持（含 v6，靠 JoinHostPort）。
- 好处：可读、可迁移（换机器改 DNS 即可）、天然双栈（一个名字挂 A + AAAA，Go 自动择优）、
  也是 v6 动态前缀的唯一现实解。

### B.3 DDNS —— 不是我们的活

家宽 v6 前缀常轮换、写死地址不可行，需把监听端的当前地址持续发布到一个域名。但 **DDNS 是
个独立、成熟、已被解决的问题**（ddclient / inadyn / 路由器内置 / cloudflare cron 一大把），
且要持有 **DNS 服务商的 API 令牌 = 另一份要管的密钥**。我们插手只会做出一个更差版本的现成
工具，还平白给「唯一城墙 = 那把 key」的安全故事引入新凭据面。故：

- **我们只做一件事：解析域名**（`net.DialTimeout("tcp", …)` 自带，等于免费）。DDNS 完全交给
  用户的任意标准客户端。
- **唯一的分内责任：每次 (重)连都重新解析域名**，不缓存已解析的 IP —— 这样 DDNS 更新后无需
  重启即可被拨号端捡到。当前结构天然满足（每次尝试都把**域名字符串**交给 Dial、由它重解析）；
  实现时**保持这一点、别引入 IP 缓存**即可。
- **部署提示**：DDNS 记录用**低 TTL**（如 60s），让前缀轮换后的新地址尽快生效。

### B.4 v6 运维要点（文档交付物，非代码）

- 防火墙默认拒入站（v6 无 NAT，但家用路由器普遍有状态防火墙挡入站，RFC 6092）→ 手动 pinhole。
- 源地址挑选：SLAAC 隐私扩展有多个地址，对外宣告**稳定的全局地址**。
- 空闲连接被中间盒踢：长轮询静默期，公网下需更激进 keepalive 保状态表。
- 公网钉死单端口（10 端口扫描是局域网发现时代特性，公网上没必要、还像探测）。

### B.5 v4 = NAT，更难，不自己解

- 家宽 v4 多在 NAT / CGNAT 后（CGNAT 连端口都没得转）。
- **不把 NAT 穿透做进工具**（那是 Tailscale/WG-DERP 级别的独立项目）。公网 v6 直连为主路；
  公网 v4 + 自己转端口 = 零代码白送；CGNAT / 无公网入站列为 out-of-scope。
- **当前 relay 是扇出、不是 NAT 方案**；真要解 reality-behind-NAT 是「反向/会合式 relay」，
  即 A 章四象限的一个组合。

## C. 密钥自管理（强随机 key）

> **密钥归「监听端」（可达的那端），不再默认归 reality。** 谁监听谁生成/持有 key、
> 谁打印连接命令给拨号方。随 A 章翻转，监听端可能是汇（mirror）。

### C.1 动机：消灭弱输入，而非拉伸

摘 VPN 后口令成为唯一城墙，弱口令在公网上是时间问题（在线爆破：每连一次试一个口令、
当前无限速；离线爆破：握手被抓则未拉伸口令可离线跑字典）。当初"WG 兜着"才不做拉伸的
理由已失效。WireGuard 从不收口令、只收 `wg genkey` 的强随机密钥——不"拉硬弱输入"，
而是"不让弱输入进来"。据此：

> **监听端默认生成一把强随机 key，拨号端直接使用。128 bit 起的 keyspace 让在线离线
> 爆破全免谈。** 此步落地后 PSK 拉伸基本可不修。

### C.2 密钥解析规则（核心）

**优先级：显式 `-k <value>` ＞ 密钥文件 ＞ 不加密。**

| 输入 | 行为 |
|---|---|
| `-k <value>`（含 env `LOCAL_MIRROR_SECRET`） | 用这个值（**显式最高优先**，与当前一致） |
| 未给 `-k`，但密钥文件存在 | 读文件、开加密（banner 显指纹） |
| 未给 `-k`，且无密钥文件 | 不加密（明文，与当前一致） |
| `--no-encrypt` | 强制关闭，即便文件存在（逃生门） |

- **显式优先于文件**（least surprise）：反过来"文件优先"会让 `-k newvalue` 被静默忽略。
- **取消 bare `-k`**：靠"没给就自动找文件"，避开 Go `flag` 可选值参数的 hack。
  "文件在就自动开加密"是往安全方向的 fail-safe，banner 显指纹、可见不黑箱。

### C.3 CLI 形态

| 命令 | 行为 |
|---|---|
| `local-mirror --gen-key` | 生成密钥文件（**拒绝覆盖**，除非 `--force`），**tty** 打印 key + 可粘贴的拨号端连接命令，然后退出 |
| `local-mirror --gen-key …`（带运行参数） | 建好文件后接着正常启动 |
| `local-mirror --show-key` | 把 key 重新**打到 tty** |
| `local-mirror -k <value> …` | 显式 key（最高优先） |
| `local-mirror --no-encrypt …` | 强制明文，即便有密钥文件 |

- `--gen-key` 性格：默认**生成+打印+退出**（像 `wg genkey`）；带运行参数才继续。
- 生成即打印**拨号端连接命令**（`local-mirror @<listener-address> ./dir -k <key>`）。
  监听端未必知道自己的公网域名/地址，`@` 段留占位符让用户填。

### C.4 密钥文件

- **路径** `<同步根>/.local-mirror/key`（与 cache.db / logs / backups / partial 同处
  工具已有的**每根状态目录**），权限 **600**（即便目录本身更宽）。拨号端对称：也把
  收到的 key 存进自己的 `.local-mirror/key`，免得每次启动都带 `-k`。
- **为何在工作目录、而非 `~/.config`**：
  - `.local-mirror` 是**强制忽略项**（config.go:32 内置默认，`LoadIgnoreList` 只增不减、
    用户 `-i` 无法移除）→ key **绝不会被同步**出去。
  - 路径由 `-p` 的 `StartPath` 解析，**不依赖 `$HOME`**——systemd / 容器里 HOME 未必正常，
    工作目录相对路径更稳。
  - **每根一把 key = 每链独立身份**，比全局共享一把更隔离，也贴合多任务（supervisor/YAML
    每 task 各自的根）。
  - 自包含、不往用户家目录拉屎——本就是工具一贯做法（所有状态一向在 `.local-mirror/`）。
- **格式** `base64(32 字节)`，`crypto/rand` 生成（约 44 字符，对齐 `wg genkey`）。
- **⚠ 别和可弃状态混淆**：`.local-mirror/` 里 cache.db / logs / partial 都**可重建**，有人
  习惯 `rm -rf .local-mirror` 重置；但 **key 对监听端不可弃**（重生成 = 所有拨号端失联）。
  故：`--gen-key` 拒绝覆盖（需 `--force`）；任何"清缓存"路径都**不得动 `key`**；文档写明
  丢失监听端 key 的恢复 = 重新生成并重发给各拨号端。

### C.5 安全细则

- **key 只对 tty**；稳态 banner 与日志**只显指纹**（key hash 前若干字节）。
- **`--gen-key` 不覆盖已有 key**（断链保护），要 `--force` 才重写。
- 随机源 `crypto/rand`；生成 key 仍过 `DerivePSK`（blake3 域分离），与手打口令共用派生路径。

## D. 升级影响 · 版本 · 产品姿态

### D.1 为什么是 2.0.0

三支柱合起来是一次**面向公网的彻底蜕变**，且带**用户可见的 CLI 词汇变更**（方向优先、
`-m` 降级为别名），符合大版本语义 → **v2.0.0**。

### D.2 各支柱的兼容性

- **C（密钥）非 flag-day、向后兼容**：仅在「未给 `-k`」时多一层文件查找；显式 `-k` 行为不变。
  生成 key 只是强随机字符串，喂同一条 `DerivePSK` + 现有协议，两端一致即互通，**无需协议 bump，
  可独立先发**。
- **A（解耦）不改报文**：属实现内部重构 + 新增传输旗；`-m` 旧用法经别名保持默认不变。
  握手互补校验优先复用 v3 已有的 `Role` 字段与可扩展位（tail-append），**尽量不做协议大版本跳**；
  若需新增字段再评估 v4。
- **B（网络）**：双栈监听是行为收敛（v4-only → v4+v6），需回归验证既有 v4 部署不受影响。

### D.3 运维观测：`--status`（已实现）

摘 VPS 后「不 ssh 上远端」留下观测盲区——推上去到底同步没同步？常驻进程把运行时
快照原子写入 `<同步根>/.local-mirror/status.json`（连接活跃时 1s、空闲 5s + 状态变化即刷），
`local-mirror --status` 是**另一个进程**只读它并渲染。**关键是拨出端在本地就知道自己的连接
状态**，无需登录远端。快照 `updated_unix` 兼做存活判据——超 3×5s 未更新即标 `○ stale`
（进程多半已停），进程崩了也能看到最后已知态。取「快照文件」而非控制 socket：不碰同步
协议、跨平台一致、崩溃可见、自包含于状态目录（可弃，删了下次自建）。见 internal/status。

**实时仪表盘（schema v2）**：终端里 `--status` 进入 1s 刷新循环（备用屏，Ctrl-C 退出），
管道/重定向时打印一次（脚本友好）。展示：进行中的文件 + 进度条 + 滚动速率、累计量、
连接细节、错误数，以及**常驻进程自采的资源占用**（CPU% 走 getrusage 跨平台；RSS/FD
在 linux 精确、darwin RSS 用峰值近似 + FD 标未知——生产端 linux 两者精确；外加 Go 堆/
goroutine）。**三种观测入口**（`--status` 是独立只读进程，绝不惊动服务；服务启动横幅也会印出
观测命令）：① `--status -p <根>` 或在同步目录里直接 `--status`（走 cwd）看单个；
② `--status --config tasks.yml` 聚合 YAML 每个任务一行；③ `--status --all` **全机发现**——
扫进程表找出所有运行中的 local-mirror（从各进程的 `-p`/cwd 推同步根、读根下的
status.json、pid 吻合才算），一屏看全,不用记路径。**发现走进程表、不建注册表**：
