## 0. 结论先行

**当前 v2 协议不建议原样冻结为 1.0。** 语义层是扎实的（变更游标、断点续传、
失联恢复都经过真机验证），但线格式层带着明显的演进债：5/15 的消息类型是死代码、
握手没有任何协商能力、错误通道是裸字符串。这些东西一旦随 1.0 冻结，以后每改
一次都是全网 flag-day。建议先做一轮**协议 v3 清理**（工作量不大，见 §7），
把「可演进的骨架」搭好再冻结。

## 1. 现状盘点

### 1.1 分层

```
TCP (tcp4，端口段 52345-52354 探测)
 └─ [可选] Noise NNpsk0（-k 口令，2 字节长度前缀分帧，透明加解密）
     └─ 消息层（12 字节头 + 消息体）
```

发现协议独立走 UDP（组播/子网广播，查询-应答式），有自己的魔数与线格式。

### 1.2 消息层线格式

头部 12 字节（BigEndian）：`Magic(4)=0xFBE322A8 | Type(2) | BodyLength(4) | Reserved(2)`。
消息体上限 64 MB。消息体为逐类型手写的二进制布局（定长字段 + 长度前缀字符串），
**唯一例外**是 TreeResponse 的 Data 字段内嵌 JSON（`[]tree.Node`）。
路径在线上统一 `/` 分隔（2026-07-15 起，编解码边界转换）。

15 个消息类型中，**5 个已是死代码**：

| 类型 | 状态 |
|---|---|
| Handshake / FileRequest / FileResponse / FileData / FileComplete / Error / TreeRequest / TreeResponse / RecentChangeRequest / RecentChangeResponse | 活跃 |
| HeartbeatPing / HeartbeatPong | 死（长轮询往返即保活） |
| Reverify / ReverifyResponse | 死（重连改用重发 Handshake） |
| Acknowledge | 死（决策：不做逐块确认，避免双向阻塞死锁） |

### 1.3 交互模型

严格的**同步请求-响应**：单连接、单飞行请求，客户端把一切串行化。
变更查询是长轮询（服务端挂起 ≤50s）；文件传输是
`FileRequest → FileResponse → FileData* → FileComplete` 的单向流，
SessionID 用于校验流完整性（错位即判连接污染、断连重建），不用于并发复用。

隐含的**跨端时序常量**（不在协议中传输，靠两端编译期一致）：
服务端挂起 50s < 客户端读超时 60s < 服务端空闲断连 90s。

### 1.4 身份与游标

- InstanceID（进程启动时随机 uint32）同时充当双向身份：客户端凭它注册进服务端
  clientMap；服务端的 ID 回显在每个变更响应里，客户端据此发现服务端悄悄重启
  （本地树缓存失效 → 全量对账）。自连接防护（中继扫到自己）也靠它。
- 变更游标 `[start, CoveredUntil]` 全程用**服务端时钟**，客户端只保存不解释；
  正确性来自区间查询（1 小时滚动日志），长轮询只是「高效闹钟」——丢通知无害。
  休眠感知（墙钟跳变 >3min）强制全量对账。这套语义是协议里最经得起推敲的部分。

## 2. 优点（值得保留的设计）

1. **串行同步模型换来的可审计性**。没有多路复用、没有请求 ID、没有并发状态机，
   任何失败的恢复动作统一是「断连重建 + 全量对账」（crash-only 风格）。
   两轮真机测试中所有连接类缺陷都能靠日志线性推理定位，这就是这个模型的红利。
2. **游标语义正确性不依赖任何单次消息送达**。区间查询 + 服务端单时钟 +
   低频全量兜底 + 休眠强制对账，四层递进。这是整个协议的地基，1.0 应原样冻结。
3. **解码纪律好**。所有 decoder 都有边界校验（body 上限 64MB、DataLength 与剩余
   字节比对、changeCount 合理性上限、discovery 严格长度校验）——都是历次审计
   换来的，恶意/损坏输入不会引发 OOM 或超量分配。
4. **失步检测与自愈**。SessionID 错位判连接污染；drainFileSession 排空过期流保住
   连接复用；重连后重发 Handshake 恢复服务端注册（曾经的真实缺陷，已修）；
   clientMap 用 CompareAndDelete 处理快速重连竞态。
5. **分层干净**。加密层对消息层完全透明（包 net.Conn）；发现协议独立魔数 +
   域分离密钥（PSK 与 discovery MAC 各自派生）；路径线格式转换只发生在编解码边界。
6. **资源卫生**。连接数信号量上限（256）、空闲断连、TCP keepalive、长轮询挂起
   有上限，服务端不会被僵尸连接拖死。
7. **流控决策成熟**。不做逐块 ACK（注释里记了双向阻塞死锁的教训）；续传靠本地
   分片大小 + 服务端指纹比对，协议零状态。

## 3. 弱点与债务

按影响排序：

### 3.1 【最重】版本策略是纯 flag-day，且中继链会放大它

握手只有 `Version(2)+UUID(4)+Role(1)`，两端**严格相等**否则拒绝。没有
min/max 支持区间、没有 feature bitmap、没有任何协商。v1→v2 已经经历过一次
flag-day；现在有了 brew/scoop/deb 分发渠道，**多台机器不会同步升级**——
尤其中继链 A→B→C，逐台升级的过程中链路必断。对单人自用工具，「全网同版本」
是可辩护的策略，但它应当是**明确决策**而非缺省；而且即使保留严格相等，
握手本身也该先变成可扩展的（见 §7），否则连「未来加协商」这条路都焊死了。

另一处不对称：版本不匹配时服务端直接断连、**不回错误消息**，旧客户端只看到
EOF/超时，无从得知原因（只有服务端日志知道）。客户端比服务端新时才能在
本地看出版本不符。混版本时的诊断体验是残缺的。

### 3.2 【重】错误通道是裸字符串

MsgTypeError 只有自由文本（服务端错误原样下发，含 OS 级原因）。没有错误码、
没有「可重试/永久」的区分、没有机器可读的路径字段。客户端只能靠单项失败
计数器等启发式补偿（permission-denied 无限重试问题的根源之一）。
一旦 1.0 冻结，错误文本会变成事实上的 API——下游一旦开始字符串匹配，
改一个措辞都是破坏性变更。

### 3.3 【中】无界响应：超大目录/变更风暴的确定性失败

单条消息体上限 64MB 是硬性线格式契约，但两类响应没有分页机制：

- **TreeResponse**：单目录全部条目一个 JSON 块。每条目约 250 字节，单目录
  ~26 万条目即触顶。触顶后是**确定性失败**（重试无效）。
- **RecentChangeResponse**：1 小时窗口内的全部变更目录一次性下发。极端场景
  （如对百万目录树 `chmod -R`）超限后，客户端断连重建 → 全量扫描 → 游标归 0 →
  下一次变更请求又拿整个窗口 → 再超限。**活锁最长持续到日志窗口滑过（1 小时）**，
  能自愈但期间同步停摆。

### 3.4 【中】死代码与残留字段占了协议面的三分之一

§1.2 的 5 个死类型之外还有：RecentChangeRequest.endTime（服务端用自己的时钟，
字段无用）、FileData.Offset（客户端盲目 append，从不校验此字段）、
FileRequest/TreeRequest 的 PathLength 由调用方设置而编码器写自己转换后的
字节（长度恰好相等才成立的脆弱不变量——wirepath 修复时依赖了 ToSlash 等长
这一巧合）、头部 Reserved 2 字节从未使用、TreeResponse.RootPath 客户端不消费。
冻结前不清掉，它们就要被永久背着。

### 3.5 【中】解码器不拒绝尾部多余字节——半个「意外扩展点」

几乎所有二进制 decoder 读完已知字段即返回，不校验缓冲区已耗尽。这意味着
**同版本内向消息体尾部追加字段是可行的**（旧端静默忽略）——但它是「意外获得」
而非设计承诺，而且被 §3.1 的版本相等检查整体锁死（旧端根本走不到解码那步）。
要么在 v3 把它扶正为正式演进机制（文档化「追加即兼容」），要么改为严格
校验拒绝尾随垃圾。现状两头不靠。

顺带：发现协议正相反——probe 定长严格校验（len≠36 即拒）、reply 长度逐字段
核算，**完全不可追加**。版本不符静默丢弃 + `-r` 直连逃生，升级期的退化行为
可接受，但要知道发现层是全协议里最刚性的一段。

### 3.6 【轻】零散健壮性瑕疵

- 未握手连接发 RecentChangeRequest：服务端查不到 clientMap 只记日志、
  **不回包也不断连**，客户端干等 60s 超时。与 Tree/File 请求（回错误并断连）
  行为不一致。
- 服务端发送路径无写超时：sendFileData 对一个停止读取的客户端可长时间阻塞，
  占住连接槽位（最终靠 keepalive 兜底，分钟级）。
- sendMessage 头与体分两次 Write：加密层下每次 Write 至少一个 Noise 帧，
  小消息固定多付一帧开销（正确性无碍，效率小疵）。
- tcp4-only（macOS 双栈问题的规避决策）+ 发现协议 IPv4 组播——IPv6 整体缺席。
  值得注意：发现应答不含 IP 字段（取 UDP 源地址），所以**线格式本身与地址族
  无关**，未来加 IPv6 不需要改协议，只需要改 socket 层。

### 3.7 【安全注记】PSK 未做口令拉伸

`DerivePSK = blake3("前缀"+secret)`——单次哈希，无 argon2/scrypt 拉伸。
NNpsk0 的首条握手消息带有由 PSK 派生密钥的 AEAD tag，**被动抓包后可离线
字典攻击口令**。当前缓解只能靠文档：`-k` 应使用长随机串而非人类口令。
未来协议修订值得引入 argon2id 拉伸（参数需进握手协商 → 又回到 §3.1）。
明文模式的信任模型（可信局域网、Role 字段不做授权）保持现状即可，但应
在 1.0 文档里白纸黑字写明。

## 4. 扩展性总评

| 层面 | 现状 | 评价 |
|---|---|---|
| 跨版本演进 | 严格相等，无协商 | **不可演进**，每次都是 flag-day |
| 同版本消息体演进 | 尾部追加被静默容忍 | 意外可行，未文档化，不可依赖 |
| 树数据（JSON 载荷） | 未知字段双向忽略 | **天然可演进**（加节点字段零成本） |
| 头部 | Reserved 2 字节未用 | 可承载 flags，但需协商配合 |
| 发现协议 | 定长严格校验 | 完全刚性，版本不符=互相不可见 |
| 传输语义（压缩/增量传输等） | 无能力位 | 任何新能力都需要先有协商机制 |

一句话：**JSON 载荷部分很软，二进制骨架很硬，而版本门闩把软的部分也锁死了。**
未来大概率想要的能力——压缩传输、rsync 式增量、多客户端游标优化、PSK 拉伸
参数——全部卡在「握手无协商」这一个瓶颈上。

## 5. 健壮性总评

抗恶意/损坏输入：**良好**（边界校验完备，历次审计的成果扎实）。
抗网络异常：**良好**（失步检测、断连重建、全量兜底、休眠感知，真机验证过）。
抗规模异常：**中等**（§3.3 的无界响应是已知确定性失败面）。
抗混版本部署：**弱**（§3.1，且诊断体验残缺）。
多客户端：结构上成立（游标在客户端手里，服务端无每客户端状态；变更广播是
channel close，天然多播），但**从未测试**——1.0 若宣称支持需先补测。

## 6. 与 1.0 的关系：冻结什么

**建议原样冻结**（这些已被验证是对的）：
- 串行请求-响应模型 + 「一连接一飞行请求」不变量
- 游标/区间查询/CoveredUntil/休眠对账语义
- 文件流无 ACK + 本地分片续传 + SessionID 失步检测
- 路径线格式 `/` 约定
- Noise NNpsk0 + 域分离派生（补口令强度文档）
- 边界校验各上限（64MB 等，作为文档化契约）

**建议冻结前先改**（即协议 v3，一次性 flag-day 把未来的 flag-day 消掉）：见 §7。

**建议明确声明为非目标**（写进 1.0 文档）：
多路复用/并发传输、压缩、增量传输、IPv6（暂缓）、跨版本互通（若 §7 未采纳）。

## 7. 协议 v3 清理清单（冻结前一次做完）

预估都是小改动，合起来一个版本号解决：

1. **握手可扩展化**：改为 `Version | MinVersion | UUID | Role | FeatureBits(8B, 全零)
   | 预留`，服务端在 `[MinVersion, Version]` 区间内择高应答。当前实现里
   区间恒为 [3,3]，行为与严格相等一致——**代价为零，但 v4 起就有协商的余地**。
2. **版本不符时回一条 Error 再断连**：让旧客户端也能打出人话（"server requires
   protocol X, you have Y"），消除 §3.1 的诊断不对称。
3. **删除 5 个死消息类型与残留字段**（Heartbeat×2、Reverify×2、Acknowledge；
   endTime、FileData.Offset 二选一——要么删要么校验；PathLength 改由编码器
   自算；TreeResponse.RootPath 删）。
4. **错误消息结构化**：`{Code uint16, Path string, Message string}`。
   码表起步只需五六个（NotFound / PermissionDenied / OutOfRoot / TooLarge /
   Internal），客户端的失败隔离从此有依据，permission-denied 无限重试可以
   按码退避。
5. **决断尾部追加语义**：推荐扶正——文档化「消息体尾部追加字段=同版本兼容
   演进」，decoder 保持忽略尾随字节（现状即是），新字段一律带存在性语义。
6. **无界响应加护栏**：TreeResponse 加续页游标（`ContinueFrom` 字段，空=完）；
   RecentChangeResponse 超阈值时降级为单条「请全量对账」信号（客户端本就有
   全量路径，复用即可）。两者都消掉 §3.3 的确定性失败。
7. （可选）**PSK 拉伸**：argon2id 参数随握手 FeatureBits 声明；不做也行，
   但 1.0 文档必须写明 `-k` 用长随机串。

以上做完再打 v1.0.0，冻结面就是一个清爽、可演进、无已知确定性失败的协议。
不做 v3 直接冻结 v2 也是一种选择——但要接受：死代码永久入契约、
错误文本成 API、下一个新能力到来时以 v1→v2 同样的方式全网翻版本。

## 8. v3 落地情况（2026-07-20）

§7 第 1-6 项全部实现，随 v0.13.0 发布，`ProtocolVersion 2→3`：

- 握手：`Version | MinVersion | UUID | Role | FeatureBits(8B)`，会话版本取两端
  `[Min, Version]` 区间交集最高值（当前恒 [3,3]）。版本不符时服务端先回一条
  `ErrCodeVersionMismatch` 错误再断连。
- 错误：`ErrorMessage{Code, Path, Message}`，六码表（Internal / NotFound /
  PermissionDenied / OutOfRoot / TooLarge / VersionMismatch）。客户端对
  PermissionDenied 按永久失败处理，提示一次后跳过——了结了 permission-denied
  无限重试。
- 分页：TreeResponse 加 `ContinueFrom` 游标，单页 20000 条；客户端循环拉全。
- 降级：RecentChangeResponse 加 `FullResync` 标志，变更数 >8192 时省略列表，
  客户端全量对账后把游标推进到 `CoveredUntil`（消掉活锁）。
- 清理：删 5 个死消息类型（编号留空洞不复用）与残留字段。
- 演进机制：尾部追加扶正为正式约定（decoder 忽略尾随字节），写入 protocol.go。

与 v2（含已发布的 0.9-0.12）不互通，握手不匹配时双向干净拒绝——已用
scoop 上的 0.12.0 真机验证。

### 8.1 §3.6 遗留瑕疵收尾（2026-07-21）

v3 未纳入 §7 清单的两条【轻】瑕疵补齐（均不改线格式、不破坏兼容）：

- **3.6b 发送路径写超时**：`sendMessage` 每次写入前置 60s 写超时
  （`sendMessageWriteTimeout`）。停止读取的对端会把 Write 挂在塞满的发送缓冲
  上，原先只能等 TCP keepalive 分钟级兜底、期间占住连接槽。这是「单次写入无
  进展」的上限，正常传输持续推进即不断刷新，不影响大文件/大分页。
- **3.6c 头体合并单帧**：`sendMessage` 把 12 字节头与消息体拼成一个缓冲一次
  写出。加密层下小消息（握手、各类请求、FileComplete）从此只付一个 Noise 帧
  而非两个，省一个 AEAD tag + 长度前缀 + 一次 Encrypt；明文下少一次系统调用。
  接收侧 `receiveMessage` 用 `io.ReadFull` 逐字段读，与发送端如何分块无关，
  故完全兼容——已验证收发两侧及加密层 `secureConn.Read` 均按流重组。

剩 3.6d（IPv6 / tcp4-only）是明确非目标，不动。

### 8.2 §3.7 PSK 拉伸：正式定为不做（WireGuard 判例）

复盘后确认 §3.7 的 argon2id/scrypt 拉伸**不做**，理由充分且成体系：

- **WireGuard 的解法是「没有这个问题」**：它压根不接受口令，密钥永远是
  `wg genkey` / `wg genpsk` 出的 32 字节高熵随机串。对「离线暴力猜密钥」的
  回答是「输入即 256 位随机，暴力破解不在桌面上」——因此不需要任何 KDF 拉伸。
  拉伸的唯一意义是补救「人类敲了弱口令」，而 WireGuard 从设计上不给敲弱口令
  的机会。
- **local-mirror 采同一姿态**：README 已明确要求 `-k` 用 `openssl rand -base64 24`
  级别的长随机串（≈144 位熵）。照做则暴力破解同样出局，argon2id 加上去零收益。
  §3.7 漏洞仅对「弱口令 + 不可信网络」同时成立。
- **代价不对称**：PSK 在 Noise 握手**之前**派生（见 §9），FeatureBits 够不着，
  只能走版本/配置约定，即一次加密模式 flag-day（且失败是不明所以的握手错误，
  连结构化诊断都发不出）。为补一个「文档已叫你别踩」的坑而中断加密链，不值。
- **结论**：1.0 姿态学 WireGuard——**要求好密钥，不拉伸烂密钥**。此项从
  §9「协商机制的天然盲区」备忘降为「明确非目标」，除非未来出现「必须支持
  人类口令」的真实需求，否则不再重提。

## 9. 1.0 之后的特性演进准则

FeatureBits 与尾部追加机制建好之后，「加特性」的默认成本从 flag-day 降到了零。
但这也意味着需要纪律，否则 64 个 bit 会被乱用。**判据：一个特性归谁管，
取决于「两端不一致时会发生什么」。**

| 不一致的后果 | 归属 | 机制 |
|---|---|---|
| 数据损坏 / 解码错位 / 死锁 | **版本约定**（硬门槛） | bump ProtocolVersion，flag-day |
| 退化到旧行为，依然完全正确 | **握手协商**（软开关） | FeatureBits 交集生效 |
| 单边理解即可，不理解就当没看见 | **尾部追加 / JSON 加字段** | 同版本内直接加 |
| 每次操作都可能不同 | **消息内标志位** | 由 FeatureBits 门控 |
| 与线上字节无关（限速、ACL、忽略规则） | **不进协议** | 本地配置 |

一句话：**改「既有对话的形状」用版本；加「可选的新对话」用协商；塞「多余的话」用追加。**

**v3 买到的关键期权**：`新消息类型 + FeatureBits 门控` 可以引入全新交互而不 bump
版本——旧端没声明该 bit，新端就永不发那些新消息类型，旧端永远见不到不认识的东西。
版本号从此只在**改动既有消息的布局或语义**时才需要动。

铁律：**被协商的特性，「关」的路径必须永远保留且永远正确**。若某特性做到「新版
必须开、旧路径准备删」，它从第一天起就该走版本，不该占 bit。

### 候选特性的归位（备忘，非承诺）

- **传输压缩**（zstd）：FeatureBits 声明支持 + FileData 帧内标志位按文件决定是否
  压缩（已压缩的媒体不值得）。两层机制配合的典型。
- **增量传输**（rsync 式 delta）：签名/delta 都是**新消息类型**，bit 门控后完全
  不动既有消息——v3 之后不再需要 flag-day。未来最值钱的特性。
- **时序参数披露**（50/60/90s）：服务端握手响应**尾部追加**自己的 LongPollHold，
  新客户端据此推导读超时，旧客户端用编译期默认。能把这三个隐式跨端常量从冻结面
  里解放出来。成本几行，是最该顺手做的一项。
- **文件元数据扩展**（Unix mode 位、mtime 精度、xattr）：tree.Node 是 JSON，
  **加字段**天然软兼容，连追加机制都不用走。
- **多路复用 / 并发传输**：要给所有消息加请求 ID，破坏「单连接单飞行」不变量——
  **v4 级别**，且长期不建议做（串行化是本协议可审计性的来源，见 §2.1）。
- **校验算法更换**（blake3 → 其它）：改既有字段语义，**必须走版本**。
- **PSK 拉伸**（argon2id）：特殊——发生在 Noise 握手**之前**，FeatureBits
  够不着，只能走版本或配置约定。这是协商机制的天然盲区。
- **不进协议**：限速（本地策略）、Role 授权（服务端本地 ACL）、IPv6（纯 socket 层）。
  发现协议扩展走它自己的版本（定长 + silent-drop 兼容模型，与 TCP 协议解耦）。

**两条反面提醒**：① 每个 FeatureBit 是永久的测试矩阵翻倍（两个 bit = 四种组合），
准入门槛是「有真实痛点 + 关闭路径零成本维护」，不是「这特性挺好」；bit 低位先用、
每个进文档、废弃的永不复用。② 「简单」是本协议目前最大的特性，不是待填的坑——
问题域（单向、可信 LAN、单操作者）撑不起也不需要 syncthing 那种复杂度。v3 买的是
「以后想加时不用翻版本」的期权，期权买完就停，等真实需求来行权。
