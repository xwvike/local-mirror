## 计划中的扩展（按优先级）

### 1. 服务自动发现
客户端 `-r` 为空时自动找到局域网内的服务端（目前为空只会回退连本机 127.0.0.1）。

- 方案：UDP 广播/组播。服务端周期广播 `魔术字 + 协议版本 + InstanceID + TCP 端口`，
  客户端启动时监听数秒，选取第一个（或列出让用户选）。
- 比 mDNS 少一个依赖，与现有自定义二进制协议风格一致。
- 落点：`internal/network/discovery.go`（此前的空占位文件已删除，实现时重建）。

### 2. 断点续传落地
协议已经预留了 `FileRequestMessage.Offset` 和 `Acknowledge` 消息，服务端也支持按 offset Seek，
但客户端 `DownloadFile` 永远从 0 开始。

- 客户端下载中断时保留临时文件（当前实现是 `.local-mirror-dl-*` 临时文件 + 校验后原子改名），
  记录已收字节数，重试时带 offset 请求剩余部分。
- 注意：续传后仍需对整个文件做 blake3 校验。

### 3. 目录树缓存跨重启持久化
`InitDB` 目前每次启动清空重建所有 bucket，大目录树启动很慢（全量 WalkDir + 逐文件哈希）。

- meta 桶里已有 `start_path`，启动时校验一致后可复用旧数据。
- 用 `size + mtime` 判断文件是否需要重新哈希，只对变化的文件重算。
- 需要处理进程离线期间发生的变更（一次启动时全量比对校准）。

### 4. 传输安全 / 认证
目前局域网内任何人连上 52345 端口就能拉取全部文件。

- 最低成本：握手消息加预共享 token（启动参数或环境变量），不匹配直接断开。
- 进阶：TLS（自签证书 + 指纹校验）。

### 5. 心跳接线
`FileClient.Ping` 和服务端 pong 处理都已实现但无人调用；
`ConnectionManager.isConnValid()` 目前恒返回 true。

- 客户端起独立的心跳定时器（注意与请求/响应共用连接的互斥问题，
  当前协议是同步请求-响应模型，心跳必须与任务串行）。
- 服务端按 `LastActiveTime` 清理僵尸连接。

## 已知限制（暂不处理，记录在案）

- **跨平台路径分隔符**：节点路径按本机 `filepath.Separator` 存储并直接在协议中传输，
  Windows 服务端 + Unix 客户端（或反之）会不兼容。协议层应统一用 `/` 传输。
- **监听端口硬编码** 52345（server、client、help 文本三处）。
- **单客户端假设**：服务端支持多连接，但变更游标语义按单客户端设计，多客户端未测试。
- **Windows watcher 未验证**：maxWatches 用默认值，fsnotify 行为未实测。
- **大目录内存**：`BuildFileTree` 把全部节点收集到内存后分批写库，超大目录树会有压力。
- **忽略列表硬编码**在 `config.IgnoreFileList`，不支持用户自定义（可考虑 .gitignore 风格配置）。
